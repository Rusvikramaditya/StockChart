Project Overview

Python scanner runs daily on Windows. Stores broad NSE tradable equity data first, then scans a selected universe. Nifty 500 remains the stable default, but small/mid/liquid/recent-listing/watchlist universes must be supported. Maintains a 7-pattern research library, but the live detector registry is pruned by backtest evidence. Stacks 4 institutional filters. Scores conviction 0–100. Outputs Carbon Ember HTML + Telegram alerts. No server. No React. No framework.

Pipeline Architecture

6-Stage Pipeline — Execution Model ▼
STAGE 1: VERIFY [sync, ~2s] │ 7 SQLite checks. @stage(critical=True) │ STAGE 2: FETCH [async I/O, ~2min bulk / ~3s daily] │ Bulk: asyncio + aiohttp + semaphore(5) + retry(3×) │ Daily: Dhan batch OHLC → chunked calls for selected universe │ STAGE 3: PRE-COMPUTE [sync, ~3s] │ ├── Market Regime (4-check, compute ONCE) │ ├── Sector RS per sector (15 calcs, cached) │ └── If regime.score ≤ 1 → BEAR → Telegram warn → exit │ STAGE 4: DETECT [ProcessPoolExecutor(8), ~12s] │ selected universe × backtest-approved detector registry │ Pass NumPy arrays (not DataFrames) — lower pickle cost │ per-detector try/except — 1 crash ≠ stock lost │ per-stock 30s timeout — 1 hang ≠ scan stuck │ STAGE 5: FILTER + SCORE [sync, ~3s] │ Only pattern hits (~50-100 stocks) │ 4 filters + RSI overlay → conviction → dedup → explanation │ STAGE 5b: CHART GENERATION [sync, ~10s] │ mplfinance candlestick + volume + RSI per scored hit │ Pattern annotations drawn (lines, shading, labels) │ Pivot/target/stop as dashed horizontal lines │ PNG per stock → output/charts/ + base64 for HTML embed │ STAGE 6: OUTPUT [sync, ~2s] ├── Jinja2 → HTML dashboard (charts embedded inline) ├── Telegram: send chart photo + caption for conviction ≥ 70 └── Daily summary TOTAL: depends on selected universe. Nifty 500 daily target stays fast; broad all_nse_equity first build is fetch-dominated and can take much longer.

Stage Decorator

Timing + error handling per stage ▼
def stage(name: str, critical: bool = True): def decorator(func): @functools.wraps(func) def wrapper(self, *args, **kwargs): log.info(f"▶ {name}") t0 = time.perf_counter() try: result = func(self, *args, **kwargs) dt = time.perf_counter() - t0 log.info(f"✅ {name} — {dt:.1f}s") self.ctx.timings[name] = round(dt, 2) return result except Exception as e: dt = time.perf_counter() - t0 log.error(f"❌ {name} FAILED — {dt:.1f}s — {e}") self.ctx.errors.append({"stage": name, "error": str(e)}) if critical: raise return wrapper return decorator # critical=True → pipeline stops (verify, fetch, pre_compute) # critical=False → pipeline continues (detect, output)

Async Fetch — Bulk Download

setup/02_download_historical.py — aiohttp + semaphore ▼
MAX_CONCURRENT = 5 # Dhan rate limit async def fetch_one(session, sem, symbol, security_id, headers): async with sem: payload = { "securityId": security_id, "exchangeSegment": "NSE_EQ", "instrument": "EQUITY", "fromDate": "2021-05-19", "toDate": "2026-05-19", "expiryCode": 0, "HistoricalDataType": "D" } for attempt in range(3): # retry 3x try: async with session.post(URL, json=payload, headers=headers) as r: if r.status == 200: return symbol, await r.json(), None if r.status == 429: await asyncio.sleep(2 ** attempt) except: await asyncio.sleep(2 ** attempt) return symbol, None, "3 retries exhausted" async def fetch_all(symbols_df, token): sem = asyncio.Semaphore(MAX_CONCURRENT) headers = {"access-token": token, "Content-Type": "application/json"} async with aiohttp.ClientSession() as session: tasks = [fetch_one(session, sem, r['Symbol'], str(int(r['sid'])), headers) for _, r in symbols_df.iterrows()] return await asyncio.gather(*tasks, return_exceptions=True)

Multiprocess Detection

ProcessPoolExecutor — 8 workers, NumPy arrays ▼
from concurrent.futures import ProcessPoolExecutor, as_completed def _scan_one_stock(args): """Worker — runs in SEPARATE PROCESS. Receives NumPy arrays.""" symbol, daily, weekly = args # daily = {"open":np,"high":np,...} results = [] for detector_fn in ALL_DETECTORS: try: hits = detector_fn(daily, weekly) results.extend(hits) except Exception: pass # 1 detector crash ≠ stock lost return {"symbol": symbol, "price": float(daily["close"][-1]), "patterns": results} @stage("Pattern Detection (selected universe)", critical=False) def detect(self): work = [(sym, loader.get_arrays(sym, "daily"), loader.get_arrays(sym, "weekly")) for sym in self.ctx.symbols] with ProcessPoolExecutor(max_workers=8) as pool: futures = {pool.submit(_scan_one_stock, w): w[0] for w in work} for f in as_completed(futures): try: r = f.result(timeout=30) # 30s per stock if r["patterns"]: self.ctx.pattern_hits.append(r) self.ctx.stocks_scanned += 1 except Exception as e: self.ctx.errors.append({"stage":"detect","symbol":futures[f]})

Resilience Strategy

FailureHandling
Dhan API down (bulk)Retry 3× per stock, exponential backoff (1s→2s→4s). Continue with available data.
Dhan batch OHLC fail (daily)Fall back to yesterday's data. Dashboard shows "as of yesterday".
1 stock crashes in detectiontry/except per stock in worker. Other selected symbols continue. Error logged.
1 detector crashes on a stocktry/except per detector inside worker. Other 6 detectors still run.
SQLite corruptCritical. Verify catches it. Pipeline aborts with clear message.
Telegram downNon-critical. Retry 2× with 5s. HTML still generates.
Sector index missingDefault NEUTRAL RS. Log warning.
Bear regimeEarly exit. Telegram: "🔴 BEAR — no breakouts." Dashboard shows regime only.
ProcessPool worker timeout30s timeout per stock. Stuck worker killed, others continue.
Golden rule: Graceful degradation at every level. One bad stock, one bad detector, one bad API call — scan continues. Only SQLite corruption or >50% data missing stops the pipeline.

This is STATIC per pattern type (same text every time, hardcoded). ASCENDING TRIANGLE: "An Ascending Triangle forms when price repeatedly hits the same resistance level (flat top) while making higher lows (rising bottom). It looks like a triangle squeezing toward a point. Why it forms: Sellers defend a fixed price level. But each time price pulls back, buyers step in earlier and at higher prices — showing increasing demand. Eventually, sellers at the resistance level get exhausted (no more supply), and price breaks out upward with force. Historical win rate: ~72-75% when confirmed with volume." CUP & HANDLE: "A Cup & Handle forms when price drops, rounds out a smooth bottom (the cup), recovers back to the starting level, then pulls back slightly (the handle) before breaking out. Why it forms: The cup is institutional accumulation — smart money quietly buying while the stock looks weak. The handle shakes out the last weak holders. When the handle completes, there's no one left to sell — and the breakout is clean. Popularized by William O'Neil (CANSLIM). Historical win rate: ~65-68%." BULL FLAG: "A Bull Flag forms after a sharp rally (the pole) followed by a tight, slightly downward-drifting consolidation (the flag). It looks like a flag on a flagpole. Why it forms: After a strong move up, the stock pauses to digest gains. Crucially, volume dries up during the flag — meaning institutions are NOT selling, just resting. When buying resumes, the next leg up mirrors the pole's size. Historical win rate: ~63-67%." VCP (Volatility Contraction Pattern): "A VCP shows a series of price swings that get progressively smaller — like a bouncing ball losing energy. For example: 25% swing → 15% → 8% → 4%. Volume dries up with each contraction. Why it forms: Each swing shakes out weaker holders. By the final tight contraction, almost everyone who wanted to sell has already sold. The remaining holders are strong hands. When new buying enters, there's no supply to absorb it — price moves fast. Created by Mark Minervini. His backtests show ~70%+ win rate in confirmed bull markets." INV HEAD & SHOULDERS: "An Inverse Head & Shoulders has three troughs — left shoulder, head (the deepest), and right shoulder. The head represents final capitulation (maximum pessimism). The right shoulder's higher low means sellers are losing power. Why it forms: It marks the END of a downtrend. The head is the last panic sell. By the right shoulder, bears can't push price as low as before. When price breaks above the neckline (peaks between the shoulders), the trend officially reverses. One of the most reliable reversal patterns. Win rate: ~74%." SUPERTREND: "Supertrend is an ATR-based trend indicator that plots a line below price in uptrends (support) and above price in downtrends (resistance). When price crosses above the Supertrend line, it flips bullish. Why it works: It adapts to volatility — wider stops in volatile markets, tighter in calm markets. A bullish flip means the stock has broken above its volatility-adjusted resistance. Settings: ATR(10) × 3.0 multiplier (default)." MULTI-YEAR BREAKOUT: "A Multi-Year Breakout happens when price finally breaks above a resistance level that has held for 2+ years. Multiple failed attempts over years create a well-known ceiling. Why it's powerful: Every trader who bought at that level over 2-5 years and was stuck — they've either sold at breakeven or given up. By the time it finally breaks, there's virtually NO overhead supply left. The move tends to be explosive and sustained. The longer the base, the bigger the breakout. Livermore, Darvas, Minervini — all loved this setup." ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ SECTION 2: THIS STOCK SPECIFICALLY — What the detector found ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Dynamic. Backtesting Engine
PENDING
▼

Answer the only question that matters: do these patterns actually make money on Indian markets? No backtest = gambling with conviction scores.

backtest/engine.py — architecture ▼
HOW IT WORKS: ━━━━━━━━━━━━ Walk through historical data day-by-day, simulating what the scanner would have detected at each point in time. For each trading day T in the lookback period: 1. Slice data up to day T (no future data leakage) 2. Run the current backtest-approved detector registry on this slice 3. Apply 4 stacking filters 4. Score conviction 5. For each pattern hit with conviction ≥ threshold: - Record ENTRY at pivot price (or next day's open) - Track forward: did price hit TARGET or STOP first? - Record outcome: WIN (hit target) / LOSS (hit stop) / OPEN (neither in N days) 6. Calculate metrics KEY RULE: NO FUTURE DATA LEAKAGE. The detector on day T must only see data up to day T. This is the #1 mistake in backtesting — if you leak even 1 day, results are garbage.
Backtest metrics — what we measure ▼
PER PATTERN: win_rate = wins / (wins + losses) avg_win_pct = average % gain on winning trades avg_loss_pct = average % loss on losing trades profit_factor = (total $ won) / (total $ lost) avg_holding_days = average bars from entry to exit max_drawdown = worst peak-to-trough during any trade expectancy = (win_rate × avg_win) - (loss_rate × avg_loss) PER CONVICTION TIER: win_rate_90plus = win rate for 90+ conviction hits win_rate_70_89 = win rate for 70-89 hits win_rate_50_69 = win rate for 50-69 hits → This proves whether conviction scoring actually works PER FILTER: win_rate_with_stage2 vs win_rate_without_stage2 win_rate_with_volume vs win_rate_without_volume win_rate_with_sector_rs vs win_rate_without_sector_rs → This proves whether each filter adds real edge COMBINED: equity_curve = cumulative P&L over time sharpe_ratio = annualized risk-adjusted return max_consecutive_losses = how many losses in a row (psychological) STACK BONUS VALIDATION: win_rate_1_pattern vs win_rate_2_stacked vs win_rate_3_stacked → Proves whether stacked patterns actually perform better
backtest/engine.py — core walkforward loop ▼
def run_backtest(db_path, universe="nifty500", lookback_years=3, entry_mode="next_open", max_hold_days=30, min_conviction=50): """ Walkforward backtest. NO future leakage. lookback_years: how far back to test (3yr = ~750 trading days) universe: profile to test separately (nifty500, small_mid_liquid, recent_listings, watchlist) entry_mode: "pivot" (enter at exact pivot) or "next_open" (realistic) max_hold_days: close trade if neither target nor stop hit in N days min_conviction: only test hits above this threshold """ loader = DataLoader(db_path) symbols = loader.get_symbols_for_universe(universe) trades = [] # Walk day by day for day_idx in range(start_idx, end_idx): current_date = trading_days[day_idx] for symbol in symbols: # Slice data UP TO current_date only daily_slice = loader.get_daily_up_to(symbol, current_date) weekly_slice = loader.get_weekly_up_to(symbol, current_date) if len(daily_slice) < 200: continue # Run detectors on sliced data daily_arrays = to_numpy_arrays(daily_slice) weekly_arrays = to_numpy_arrays(weekly_slice) for detector in ALL_DETECTORS: hits = detector(daily_arrays, weekly_arrays) for hit in hits: if hit.status not in ["BREAKING OUT", "PIVOT READY"]: continue # Score it score = score_conviction(hit, ...) if score < min_conviction: continue # Record entry if entry_mode == "next_open": entry_price = get_next_open(symbol, current_date) else: entry_price = hit.pivot # Track forward (using future data — this is allowed for OUTCOME only) outcome = track_trade_forward( symbol, current_date, entry_price, hit.target, hit.stop_loss, max_hold_days ) trades.append({ "symbol": symbol, "entry_date": current_date, "pattern": hit.pattern, "conviction": score, "entry_price": entry_price, "target": hit.target, "stop_loss": hit.stop_loss, **outcome, }) return BacktestResult(trades) def track_trade_forward(symbol, entry_date, entry, target, stop, max_days): """Check each subsequent day: did price hit target or stop?""" for i in range(1, max_days + 1): day_data = get_day(symbol, entry_date + i) if day_data is None: continue if day_data.low <= stop: return {"result": "LOSS", "exit_price": stop, "hold_days": i} if day_data.high >= target: return {"result": "WIN", "exit_price": target, "hold_days": i} # Neither hit — close at last price return {"result": "TIMEOUT", "exit_price": last_close, "hold_days": max_days}
